# Incident Investigation Kit

A Claude-powered incident investigation tool for the Cue team.

## Quick Start

1. Place this repo as a sibling to your `cue`, `acl`, and `cqs` repos
2. Generate a Dynatrace platform token (see `.env.example` for required scopes and instructions)
3. Run `./setup.sh` in your terminal to configure API tokens
4. **Cut a branch for the investigation** — e.g. `git checkout -b investigation/INC11638105`
5. Open this directory in Claude Code (`claude`) — it will guide the rest
6. When the investigation is complete, open a PR to merge the `log/` entry into `main`

```
    └── YYYY-MM-DD-*/
```

## Investigation Log

`log/` holds one lightweight summary file per incident — committed to the repo, no sensitive data. Each file covers the date, incident ID, what was affected, root cause, and outcome in a few lines.

`.active-investigations/` is where Claude keeps raw session data during an investigation (queries, log dumps, scratch notes). It's gitignored. At the end of an investigation, the useful parts get distilled into a `log/` entry.

This separation keeps sensitive log data out of the repo while building a committed history that can be published as a status or incident history page when the team is ready.

## Architecture Docs

The `architecture/` folder is the source of truth for what services Cue, ACL, and CQS actually call. Claude reads these at the start of each investigation before touching any logs.

**Keep additions here minimal and verified** — a lean, accurate architecture doc is far more useful than a comprehensive one with stale entries. Every investigation that uncovers a new or removed dependency should update these files so the next investigator doesn't have to rediscover it.
